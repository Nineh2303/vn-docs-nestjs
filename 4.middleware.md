---

## 4.2 Middleware là gì?

Middleware là một hàm hoặc class được thực thi trước khi Request tới Route Handler.

Middleware cho phép:

- Đọc Request.
- Chỉnh sửa Request.
- Thực hiện kiểm tra ban đầu.
- Ghi log.
- Thực hiện các tác vụ tiền xử lý.

Ví dụ:

```text
Client
   ↓
Middleware
   ↓
Controller
   ↓
Provider
   ↓
Database
```

Middleware được thực thi trước Controller.

---

## 4.3 Middleware trong Request Lifecycle

Một Request trong NestJS thường đi qua:

```text
Client
   ↓
Middleware
   ↓
Guards
   ↓
Interceptors (Before)
   ↓
Pipes
   ↓
Controller
   ↓
Provider
   ↓
Interceptors (After)
   ↓
Response
```

Middleware là thành phần đầu tiên có thể can thiệp vào Request.

Điều này khiến Middleware phù hợp cho các tác vụ mang tính toàn cục.

Ví dụ:

- Logging
- Correlation ID
- Request Tracking
- Security Headers

## 4.4 Tạo Middleware

Nest CLI hỗ trợ tạo Middleware.

```bash
nest generate middleware logger
```

Hoặc:

```bash
nest g mi logger
```

Kết quả:

```text
src/
└── logger/
    └── logger.middleware.ts
```

---

## 4.5 Middleware đầu tiên

Ví dụ:

```typescript
import {
  Injectable,
  NestMiddleware
} from '@nestjs/common';

import {
  Request,
  Response,
  NextFunction
} from 'express';

@Injectable()
export class LoggerMiddleware
  implements NestMiddleware {

  use(
    req: Request,
    res: Response,
    next: NextFunction
  ) {

    console.log(req.method, req.originalUrl);

    next();
  }

}
```

---

## 4.6 Phương thức use()

Mọi Middleware đều phải triển khai:

```typescript
use()
```

Ví dụ:

```typescript
use(
  req: Request,
  res: Response,
  next: NextFunction
) {

  next();

}
```

Ba tham số:

| Tham số | Mô tả |
|----------|----------|
| req | Request hiện tại |
| res | Response hiện tại |
| next | Chuyển Request tới Middleware tiếp theo |

---

## 4.7 Vai trò của next()

Middleware phải gọi:

```typescript
next()
```

để Request tiếp tục được xử lý.

Ví dụ:

```typescript
use(
  req: Request,
  res: Response,
  next: NextFunction
) {

  console.log('Request');

  next();

}
```

Nếu không gọi:

```typescript
next()
```

Request sẽ bị treo.

Ví dụ sai:

```typescript
use(
  req: Request,
  res: Response,
  next: NextFunction
) {

  console.log('Blocked');

}
```

Client sẽ không nhận được Response.

## 4.8 Đăng ký Middleware

Middleware không tự động hoạt động sau khi tạo.

Cần đăng ký trong Module.

Ví dụ:

```typescript
export class AppModule
  implements NestModule {

  configure(
    consumer: MiddlewareConsumer
  ) {

    consumer
      .apply(LoggerMiddleware)
      .forRoutes('*');

  }

}
```

Sau đó Middleware sẽ được áp dụng cho toàn bộ Route.

---

## 4.9 MiddlewareConsumer

NestJS cung cấp:

```typescript
MiddlewareConsumer
```

để cấu hình Middleware.

Ví dụ:

```typescript
configure(
  consumer: MiddlewareConsumer
) {}
```

Consumer cho phép:

- Apply Middleware
- Chọn Route
- Loại trừ Route

## 4.10 Áp dụng cho Route cụ thể

Ví dụ:

```typescript
consumer

```

```http

```

Ví dụ:



## 4.13 Exclude Routes

Ví dụ:

```typescript
consumer

```

## 4.14 Functional Middleware

Ngoài Class Middleware, NestJS hỗ trợ Function Middleware.

Ví dụ:

```typescript
import {
  Request,
  Response,
  NextFunction
} from 'express';

export function logger(
  req: Request,
  res: Response,
  next: NextFunction
) {

  console.log(req.method);

  next();

}
```

Đăng ký:

```typescript
consumer
  .apply(logger)
  .forRoutes('*');
```

## 4.15 Logging Middleware

Một use case phổ biến:

```typescript
@Injectable()
export class LoggerMiddleware
  implements NestMiddleware {

  use(
    req: Request,
    res: Response,
    next: NextFunction
  ) {

    console.log(
      `[${req.method}] ${req.originalUrl}`
    );

    next();

  }

}
```

Ví dụ log:

```text
[GET] /users
[POST] /orders
```

## 4.16 Request Timing Middleware

Đo thời gian xử lý Request.

```typescript
use(
  req: Request,
  res: Response,
  next: NextFunction
) {

  const start = Date.now();

  res.on('finish', () => {

    const duration =
      Date.now() - start;

    console.log(duration);

  });

  next();

}
```

## 4.17 Authentication Middleware

Middleware có thể đọc Token.

```typescript
use(
  req: Request,
  res: Response,
  next: NextFunction
) {

  const token =
    req.headers.authorization;

  console.log(token);

  next();

}
```

Tuy nhiên việc phân quyền thường phù hợp với Guard hơn.

Middleware chỉ nên thực hiện kiểm tra sơ bộ.

## 4.18 Khi nào nên dùng Middleware?

Middleware phù hợp cho:

- Logging
- Request Tracking
- Correlation ID
- Security Headers
- Request Metrics
- Compression
- Cookie Parsing

Ví dụ:

```text
Request
   ↓
Middleware
   ↓
Controller
```

---

## 4.19 Khi nào KHÔNG nên dùng Middleware?

Không nên dùng Middleware cho:

- Authorization
- Role Checking
- Validation
- Business Logic

Những trường hợp này phù hợp hơn với:

```text
Authorization → Guard

Validation → Pipe

Response Transform → Interceptor

Business Logic → Provider
```

## 4.20 Best Practices

### Middleware nên đơn giản

Middleware chỉ nên xử lý một nhiệm vụ duy nhất.

Ví dụ:

```text
LoggerMiddleware
```

thay vì:

```text
Logger + Auth + Validation + Cache
```

---

### Luôn gọi next()

Ví dụ:

```typescript
next();
```

Nếu bỏ quên, Request sẽ bị treo.

---

### Không đặt Business Logic trong Middleware

Không nên:

```typescript
query database
send email
create order
```

trong Middleware.

---

### Tái sử dụng Middleware

Ưu tiên tạo Middleware có thể dùng ở nhiều Module.

---

## 4.21 Common Mistakes

### Quên đăng ký Middleware

Sai:

```typescript
@Injectable()
export class LoggerMiddleware
  implements NestMiddleware {}
```

Không cấu hình trong:

```typescript
configure()
```

Middleware sẽ không chạy.

---

### Quên gọi next()

Sai:

```typescript
use(req, res, next) {

  console.log('Hello');

}
```

Request sẽ bị treo.

---

### Sử dụng Middleware để phân quyền

Sai:

```typescript
if (!isAdmin) {
  throw ...
}
```

Nên sử dụng Guard.
