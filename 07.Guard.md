Sau khi hoàn thành chương này, bạn sẽ hiểu:

- Cách tạo Guard.

- Cách sử dụng Metadata và Reflector.
- Best Practices khi xây dựng Guard.

---

## 7.2 Guard là gì?

Guard là thành phần quyết định một Request có được phép tiếp tục xử lý hay không.

Guard hoạt động như một cổng kiểm tra quyền truy cập.

Ví dụ:

```text
Client
   ↓
Guard
   ↓
Controller
```

Nếu Guard cho phép:

```typescript
true
```

Request tiếp tục.

Nếu Guard từ chối:

```typescript
false
```

Request bị chặn.

---

## 7.3 Guard trong Request Lifecycle

Một Request thường đi qua:

```text
Client
   ↓
Middleware
   ↓
Guards
   ↓
Interceptors (Before)
   ↓
Pipes
   ↓
Controller
   ↓
Provider
   ↓
Interceptors (After)
   ↓
Response
```

Guard chạy sau Middleware và trước Controller.

---

## 7.4 Guard dùng để làm gì?

Các trường hợp phổ biến:

### Authentication

Kiểm tra người dùng đã đăng nhập chưa.

Ví dụ:

```http
Authorization: Bearer token
```

---

### Authorization

Kiểm tra người dùng có quyền truy cập hay không.

Ví dụ:

```text
Admin
User
Manager
```

---

### Feature Access

Kiểm tra tính năng được bật hay chưa.

Ví dụ:

```text
Premium Feature
Beta Feature
```

---

## 7.5 Tại sao không dùng Middleware?

Middleware có thể đọc Request.

Tuy nhiên Middleware không biết Route hiện tại yêu cầu quyền gì.

Ví dụ:

```typescript
GET /users
```

và

```typescript
DELETE /users
```

có thể yêu cầu quyền khác nhau.

Guard được thiết kế để giải quyết vấn đề này.

## 7.7 CanActivate Interface

Mọi Guard đều triển khai:

```typescript
CanActivate
```

Ví dụ:

```typescript
import {
  CanActivate,
  ExecutionContext
} from '@nestjs/common';

export class AuthGuard
  implements CanActivate {

  canActivate(
    context: ExecutionContext
  ): boolean {

    return true;

  }

}
```

---

## 7.8 Phương thức canActivate()

Đây là phương thức quan trọng nhất của Guard.

Ví dụ:

```typescript
canActivate(
  context: ExecutionContext
): boolean {

  return true;

}
```

Nếu:

```typescript
true
```

Request tiếp tục.

Nếu:

```typescript
false
```

NestJS trả:

```http
403 Forbidden
```

---

## 7.9 Guard đơn giản đầu tiên

Ví dụ:

```typescript
@Injectable()
export class AuthGuard
  implements CanActivate {

  canActivate(
    context: ExecutionContext
  ): boolean {

    return false;

  }

}
```

Mọi Request sẽ bị từ chối.

Response:

```http
403 Forbidden
```

## 7.10 Áp dụng Guard cho Route

Ví dụ:

```typescript
@Get()
@UseGuards(AuthGuard)
findAll() {}
```

Request sẽ phải đi qua:

```typescript
AuthGuard
```

trước khi tới Controller.

---

## 7.11 Áp dụng Guard cho Controller

Ví dụ:

```typescript
@Controller('users')
@UseGuards(AuthGuard)
export class UsersController {}
```

Tất cả Endpoint trong Controller đều sử dụng Guard.

## 7.13 ExecutionContext

```typescript
ExecutionContext
```

Ví dụ:

```typescript
canActivate(
  context: ExecutionContext
)
```

---

const request =

## 7.14 Đọc Header trong Guard

Ví dụ:

```typescript
canActivate(
  context: ExecutionContext
): boolean {

  const request =
    context
      .switchToHttp()
      .getRequest();

  const token =
    request.headers.authorization;

  return !!token;

}
```

Nếu không có token:

```http
403 Forbidden
```

---

## 7.15 Authentication Guard

Ví dụ:

```typescript
@Injectable()
export class AuthGuard
  implements CanActivate {

  canActivate(
    context: ExecutionContext
  ): boolean {

    const request =
      context
        .switchToHttp()
        .getRequest();

    const token =
      request.headers.authorization;

    return !!token;

  }

}
```

Request:

```http
GET /users
Authorization: Bearer abc
```

Được phép truy cập.

## 7.16 Authorization là gì?

Authentication trả lời câu hỏi:

```text
Bạn là ai?
```

Authorization trả lời câu hỏi:

```text
Bạn được phép làm gì?
```

Ví dụ:

```text
Admin
Editor
User
```

## 7.17 Role-based Authorization

Giả sử:

```typescript
request.user = {
  id: 1,
  role: 'admin'
};
```

Guard:

```typescript
canActivate(
  context: ExecutionContext
): boolean {

  const request =
    context
      .switchToHttp()
      .getRequest();

  return request.user.role === 'admin';

}
```

## 7.20 Tạo Decorator Roles

Ví dụ:

```typescript
import { SetMetadata }
from '@nestjs/common';

export const Roles =
  (...roles: string[]) =>
    SetMetadata(
      'roles',
      roles
    );
```

Sử dụng:

```typescript
@Roles('admin')
```

---

## 7.21 Roles Guard

Ví dụ:

```typescript
@Injectable()
export class RolesGuard
  implements CanActivate {

  constructor(
    private reflector: Reflector
  ) {}

  canActivate(
    context: ExecutionContext
  ): boolean {

    const roles =
      this.reflector.get(
        'roles',
        context.getHandler()
      );

    if (!roles) {
      return true;
    }

    const request =
      context
        .switchToHttp()
        .getRequest();

    return roles.includes(
      request.user.role
    );

  }

}
```

## 7.22 Guard trả Exception

Thay vì:

```typescript
return false;
```

có thể:

```typescript
throw new ForbiddenException();
```

Điều này cho phép tùy chỉnh lỗi.

## 7.25 APP_GUARD

Một cách phổ biến để đăng ký Global Guard:

```typescript
@Module({
  providers: [
    {
      provide: APP_GUARD,
      useClass: AuthGuard
    }
  ]
})
```

Ưu điểm:

- Hỗ trợ Dependency Injection.
- Có thể inject Service.

## 7.26 Best Practices

### Authentication bằng Guard

Không nên:

```typescript
if (!token)
```

trong Controller.

Nên sử dụng Guard.

---

### Authorization bằng Guard

Role checking nên nằm trong Guard.

Không nên đặt trong Service hoặc Controller.

---

### Sử dụng Decorator tùy chỉnh

Tốt:

```typescript
@Roles('admin')
```

Thay vì:

```typescript
@SetMetadata(...)
```

ở mọi nơi.

---

### Tách Authentication và Authorization

Không nên:

```typescript
AuthAndRoleGuard
```

quá lớn.

Nên:

```typescript
AuthGuard
RolesGuard
```

riêng biệt.

---

## 7.27 Common Mistakes

### Dùng Middleware để phân quyền

Sai:

```typescript
if (role !== 'admin')
```

trong Middleware.

---

### Đặt Role Logic trong Controller

Sai:

```typescript
if (
  request.user.role
  !== 'admin'
)
```

trong Controller.

---

### Tạo Global Guard không cần thiết

Không phải mọi Guard đều nên Global.

Một số Guard chỉ nên áp dụng cho các Route đặc biệt.

---

### Không sử dụng Metadata

Điều này khiến Guard khó tái sử dụng.

---

## 7.28 So sánh Middleware và Guard

| Middleware | Guard |
|------------|--------|
| Chạy trước Guard | Chạy sau Middleware |
| Không biết Route Metadata | Biết Route Metadata |
| Logging | Authorization |
| Request Tracking | Authentication |
| Correlation ID | Role Checking |
